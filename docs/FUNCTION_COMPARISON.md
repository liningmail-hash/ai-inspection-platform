# 功能比对与缺口分析

本文档基于无人机 AI 巡检平台完整功能清单，对当前仓库的一期工程骨架进行差距分析。状态分为：

- `已覆盖`：当前已有页面、接口或工程骨架。
- `部分覆盖`：已有入口或接口雏形，但还不是完整生产能力。
- `未覆盖`：当前尚未实现。
- `一期建议`：建议在近期闭环中补齐。
- `二期/三期`：依赖真实设备、三维数据、复杂算法或工程规模，建议后置。

## 总体结论

当前系统完成的是“AI巡检平台一期骨架”，不是完整无人机机场巡检商业系统。当前优势是架构清晰、前后端可预览、接口有雏形、Docker/Ubuntu 部署有基线。主要缺口集中在：

- 真实视频接入与直播播放。
- 无人机远程控制和机场控制。
- 地图、三维、航线规划。
- 媒体资产和巡检报告。
- AI算法训练、部署、监控的完整工程化。
- 用户权限、组织、审计。
- API网关、消息队列、监控日志等生产组件。

## 技术方案比对

| 能力 | 参考方案 | 当前方案 | 建议 |
|---|---|---|---|
| 前端 | Vue + Element UI | Vue 3 + TS + Vite + 自定义工业风 | 保持 Vue 3，后续可引入组件库但不急。 |
| API网关 | Spring Cloud Gateway | 未覆盖 | 二期多服务拆分后加入。 |
| 后端微服务 | 多 Spring Boot 服务 | 单 Spring Boot 中心后端 | 一期保持单体模块化，避免过早微服务。 |
| 数据库 | MySQL + MongoDB | PostgreSQL + MinIO | PostgreSQL 更适合结构化和空间扩展，MongoDB暂不需要。 |
| 消息队列 | RabbitMQ | Redis/MQTT 预留 | 云边通信优先 MQTT，业务异步可 RabbitMQ/Redis Stream。 |
| AI框架 | TensorFlow | PyTorch/ONNX/TensorRT 预留 | 工业推理更建议 ONNX/TensorRT。 |
| 文件存储 | MinIO | MinIO | 保持。 |
| 视频传输 | WebRTC/H.265 | 未接入真实视频服务 | 一期建议加入 ZLMediaKit 或 SRS。 |
| 无人机协议 | DJI SDK/MavLink | DroneAdapter 预留 | 二期按厂家接入。 |
| 三维可视化 | Cesium.js | 未覆盖 | 二期/三期加入。 |
| 地图服务 | 高德/百度 | 未覆盖 | 一期先加二维地图，三维后置。 |
| 实时计算 | Spark Streaming | 未覆盖 | 暂不建议引入，先用事件流/消息队列。 |
| 部署 | Docker/Kubernetes | Docker Compose | 一期 Compose，二期 K8s。 |
| 安全 | OAuth2/JWT | 未覆盖 | 一期必须补 JWT/RBAC。 |

## 一期补齐优先级

### P0：必须补齐，否则不像完整平台

1. 登录、用户、角色、组织、审计日志。
2. 视频服务组件：RTSP 接入、HLS/WebRTC 播放、截图。
3. 媒体资产：照片、视频、证据、报告。
4. 任务管理：即时任务、定时任务、循环任务。
5. 算法管理：算法列表、参数、模型版本、性能指标。
6. 事件地图：二维地图展示事件、设备、机场。

### P1：增强平台专业度

1. 航线导入导出：KML/CSV。
2. 飞行记录和任务回放。
3. 机场/无人机档案。
4. 边缘节点心跳、任务下发、事件补传。
5. MQTT 消息通道。

### P2：依赖真实设备或高级数据

1. 大疆机场上云。
2. MavLink/DJI 远程控制。
3. 实景三维、倾斜摄影、正射图。
4. 算法自主飞行控制。
5. Kubernetes、网关、完整微服务化。
