# 云端共享测试环境协作规则

更新时间：2026-06-16

## 1. 目标

从现在开始，`114.67.114.201` 这台云服务器作为团队共享测试环境。

用途：

- 前后端联调
- QA 验收
- 设备管理、视频、AI、告警闭环验证
- 后续持续迭代验证

原则：

- 不再反复用聊天转述“本地做好了再上传”。
- 默认以云端环境为主验证结果。
- 本地主要用于写代码、跑构建、做快速自测。
- 云端环境是当前唯一联调真相源。

## 2. 协作方式

团队成员分工：

- 程序员 A：后端代码修改后，提交到仓库，并在云端部署 backend 更新。
- 程序员 B：前端代码修改后，提交到仓库，并在云端部署 frontend 更新。
- QA：直接以云端页面和云端接口做验收，不再以本地环境为主。
- 机动人员：负责云端部署支持、权限、容器、脚本、迁移、回滚记录。

## 3. 云端环境定位

当前云端环境是：

- 共享测试环境
- 联调环境
- 准生产前验证环境

它不是正式生产环境，但它必须尽量保持可连续使用，不能被随意清空。

禁止：

- 随意删库
- 随意删 volume
- 未经总指挥允许执行 `docker compose down -v`
- 未经确认覆盖 `deploy/.env`

## 4. 推荐流程

### 后端

1. 本地开发
2. 本地 `.\mvnw.cmd test`
3. 提交代码
4. 云端重新构建并部署 backend
5. 云端接口验证
6. 回填 `docs/TASK_FEEDBACK.md`

### 前端

1. 本地开发
2. 本地 `npm run build`
3. 提交代码
4. 云端重新构建并部署 frontend
5. 打开云端页面验证
6. 回填 `docs/TASK_FEEDBACK.md`

### QA

1. 直接访问云端页面
2. 直接调用云端接口
3. 记录缺陷
4. 回填 `docs/TASK_FEEDBACK.md`

## 5. Git 目标

最终目标不是只把东西跑在服务器上，而是：

1. 本地代码持续整理
2. 代码进入 Git
3. 云端环境持续部署最新版本
4. 每修一轮就更新云端并验收
5. 最终形成可正式发布、可持续迭代的系统

本地主要负责开发和预检查，最终结果必须在云端验证。
任务看 docs/TASK_DISPATCH.md，命令看 docs/EXECUTION_COMMANDS.md，回报写 docs/TASK_FEEDBACK.md。
