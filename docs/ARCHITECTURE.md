# 架构设计

## 一期目标

一期按园区/工厂小型试点设计，支持 50-150 路摄像头、1 套无人机机场/无人机、10-20 路并发 AI 分析。架构采用私有云边协同：中心平台负责统一管理、数据闭环、训练发布和告警处置；边缘节点负责视频接入、转码、推理和断网缓存。

## 服务边界

- 前端工作台：统一承载驾驶舱、视频监控、AI巡检、告警、训练、模型、无人机、设备管理。
- 中心后端：负责业务主数据、权限审计、巡检计划、告警流转、模型发布、设备/无人机适配抽象。
- AI 服务：负责样本管理、训练任务、模型版本、推理接口，后续接入 PyTorch、ONNX Runtime、TensorRT。
- 边缘节点：负责设备协议适配、视频流、AI推理运行、事件缓存与补传。
- 数据底座：PostgreSQL 存结构化数据，MinIO 存证据图片/短视频/训练样本，Redis/MQTT 存状态和任务消息。

## 核心流程

1. 设备或无人机通过适配器注册到中心平台。
2. 运维人员创建巡检计划，选择点位、算法、ROI、时段和告警规则。
3. 中心平台向边缘节点下发推理任务。
4. 边缘节点拉流、推理、生成事件证据，并上报告警。
5. 告警中心完成确认、误报、派单和关闭。
6. 误报/漏报样本进入标注训练闭环。
7. AI服务训练新模型并登记模型版本。
8. 管理员灰度发布模型到边缘节点，必要时回滚。

## 适配器设计

`DeviceAdapter` 统一摄像头协议能力：注册、心跳、拉流、抓图、PTZ、录像查询、状态同步。海康、大华、ONVIF、RTSP、GB/T 28181、JT/T 1078 都作为具体适配器实现。

`DroneAdapter` 统一无人机和机场能力：机场状态、无人机状态、航线下发、任务启动/停止、遥测数据、视频流地址、巡检结果回传。

## 后端分层

中心后端采用清晰分层，避免业务堆积：

- `api/`：HTTP Controller，只处理请求、响应和状态码。
- `service/`：业务查询和流程编排，例如 `InspectionQueryService`。
- `repository/`：数据访问抽象，例如 `InspectionRepository`。
- `domain/`：领域对象和接口返回模型。
- `adapter/`：设备、无人机和外部系统适配接口。

当前本地默认使用 `InMemoryInspectionRepository` 提供演示数据。Docker Compose 会启用 `postgres` Profile，使用 `PostgresInspectionRepository` 从 PostgreSQL 读取数据，并保持 Controller 和前端接口不变。
